# 揭棋 AI 博弈程序 PRD

## 1. 背景与目标

本 PRD 基于课程文档《2026大作业——揭棋.docx》整理，供后续 agents 进行需求分析、架构设计、任务拆分、编码、测试和实验报告撰写时引用。

本项目目标是实现一个揭棋 AI 博弈程序。程序需要理解揭棋规则，能维护棋局状态、生成合法走法、评估暗子概率价值，并在限定时间内输出下一步走法。程序可以借用真人对弈部分的界面或协议进行检测，也可以不提供图形界面，仅通过命令行、脚本、Socket 或统一接口完成走子信息的发送与接收。

优先级最高的交付物不是精美界面，而是一个可对弈、可解释、可测试、可复现实验结果的 AI agent。

## 2. 用户与使用场景

### 2.1 主要用户

- 开发成员：根据 PRD 拆分模块、实现规则和 AI 算法。
- 测试成员：根据验收标准验证合法走法、暗子概率、搜索决策和超时控制。
- 报告撰写成员：引用需求、架构、算法、测试和 AI 辅助过程。
- 对战组织者：让本组 AI 与其他小组程序或人工棋手进行对弈。

### 2.2 核心场景

1. 开始一局揭棋对弈，初始化棋盘、明暗状态和双方棋子池。
2. 对方输入一步走法后，程序更新棋局并校验合法性。
3. AI 在当前局面和时间限制内搜索候选走法。
4. AI 对暗子使用剩余棋子池计算数学期望，纳入局面评估。
5. AI 输出一步 `Move`，供控制台、Socket 客户端或测试框架执行。
6. 程序记录棋谱、搜索日志和关键评估信息，用于调试与实验报告。

## 3. 范围

### 3.1 必须实现

- 完整棋盘状态模型：10 行、9 列，坐标采用列 `a` 到 `i`、行 `9` 到 `0` 的表示方式。
- 揭棋明暗机制：将帅开局明置，其余棋子暗置；暗子第一次移动、吃子或被主动翻面后确定真实类型。
- 合法走法生成与校验：按揭棋规则处理明子、暗子、吃子、翻面、将帅相关胜负。
- AI 决策接口：输入棋局状态，输出合法 `Move`。
- 暗子概率评估：根据己方和对方剩余未揭示棋子池，对暗子价值使用数学期望。
- 搜索算法：基础版本至少实现带时间控制的 Alpha-Beta 搜索或等价博弈搜索。
- 超时控制：AI 每步在限定时间内返回走法；建议默认每步 60 秒，测试环境可配置为更短时间。
- 棋谱与日志：记录每步 source、destination、type、时间戳、是否翻子、是否吃子、搜索深度和评估分。
- 测试用例：覆盖规则引擎、暗子概率、评估函数、搜索超时、整局对弈流程。
- 面向对象设计：至少包含 5 个以上领域类。

### 3.2 可选实现

- Socket 适配器，兼容真人对弈部分服务器的 `Move` 协议。
- 简单 GUI，用于观察棋盘和调试 AI。
- 同一程序内支持 AI vs AI、AI vs 人、AI vs 远程客户端。
- 迭代加深、置换表、走法排序、杀棋检测、开局策略、残局策略。
- 公开对战记录导出和胜率统计。

### 3.3 不在本 PRD 范围内

- 精美图形界面和复杂动画。
- 大规模联网匹配系统。
- 用户账号、排行榜、持久化数据库。
- 未经老师认可的规则扩展。

## 4. 规则需求

### 4.1 棋盘与坐标

- 棋盘为中国象棋标准 10x9 棋盘。
- 从上到下行坐标为 `9` 到 `0`。
- 从左到右列坐标为 `a` 到 `i`。
- 坐标示例：`b3`、`e0`、`h9`。
- AI 内部可以使用数组坐标，但所有外部接口必须能转换为文档要求的棋谱坐标。

### 4.2 棋子类型

`type` 编码必须兼容课程文档：

| type | 棋子 |
| --- | --- |
| 0 | 将/帅 |
| 1 | 车 |
| 2 | 马 |
| 3 | 炮 |
| 4 | 兵/卒 |
| 5 | 士/仕 |
| 6 | 象/相 |

棋子还必须包含阵营、是否明子、当前位置、真实类型、当前位置对应的原始位置类型等信息。

### 4.3 明暗与翻子

- 开局时，双方将/帅为明子，位于各自九宫原位。
- 其余 15 枚己方棋子均为暗子，随机打乱后摆在己方常规象棋初始位置。
- 暗子移动时，移动规则按其当前所在位置对应的常规象棋棋子规则判断。
- 暗子移动或吃子完成后，必须立即翻开并确定真实类型。
- 玩家也可以选择一枚己方暗子原地翻开，该操作消耗一回合。
- 吃掉对方暗子时，吃子方知道被吃棋子的真实类型；被吃方可以不知道，但程序内部需要能维护真实状态。

### 4.4 明子走法

- 明子按真实类型走法行动。
- 车、马、炮、兵/卒、将/帅遵循中国象棋基础规则。
- 马腿和象眼规则有效。
- 明子士/仕可以离开九宫并过河，仍按斜走一格移动和吃子。
- 明子象/相可以过河，仍按田字格移动，且受象眼限制。

### 4.5 胜负与和棋

- 被将死、困毙、认输或超时的一方判负。
- 文档说明“不考虑不应将这种错误”：如果一方走出使己方将帅可被直接吃掉的局面，对方可以直接吃掉将帅获胜；服务器或规则引擎不需要自动判该类错误。
- 双方连续 40 步棋无吃子，自动判和。这里的“步”按一次落子或翻面动作计数；如果老师要求按完整回合计数，应在 `RULES.md` 中改为统一口径。
- 长将、长捉可按主流规则最多 6 回合处理；兵卒长捉判和。
- 如果长将/长捉实现成本过高，第一版必须至少记录重复局面，报告中说明限制，并在增强版补足。

## 5. 数据模型

### 5.1 Move

`Move` 是 AI、测试框架和可选 Socket 适配器之间的核心通信对象。

```java
class Move {
    private String source;        // 例如 "b3"，翻面操作时可等于 destination
    private String destination;   // 例如 "b4"
    private Integer type;         // 首次翻开后的真实类型；非翻子步可为空
    private long turnStartTime;   // 回合开始时间戳，服务端或裁判时间为准
}
```

实现时建议补充内部字段，但外部协议应至少能映射以上字段：

- `moveKind`：普通移动、吃子、原地翻面、认输。
- `side`：红方或黑方。
- `clientSendTime`、`serverReceiveTime`：用于日志和超时诊断。
- `capturedPieceType`：内部日志字段，不一定对外公开。

### 5.2 推荐领域类

至少需要下列领域类中的 5 个，推荐全部建立清晰边界：

- `GameState`：完整局面、当前行棋方、历史记录、剩余时间、胜负状态。
- `Board`：棋盘格子、坐标转换、棋子查询和移动。
- `Piece`：阵营、明暗、真实类型、当前位置、初始位置类型。
- `Move`：走子、翻面、吃子和时间戳信息。
- `RuleEngine`：合法走法生成、走法校验、胜负与和棋判定。
- `HiddenPiecePool`：未揭示棋子池、概率分布、翻子结果更新。
- `Evaluator`：局面静态评估，包含暗子期望值。
- `SearchEngine`：Alpha-Beta、迭代加深、超时中断。
- `AiAgent`：对外决策入口，组织规则、评估和搜索。
- `GameRecorder`：棋谱、日志、复盘数据导出。
- `OpponentAdapter`：命令行、Socket 或 GUI 的输入输出适配。

## 6. AI 决策需求

### 6.1 Agent 输入

AI agent 每次决策需要接收：

- 当前 `GameState`。
- 己方阵营。
- 本回合开始时间。
- 本步最大思考时长。
- 可选对手历史走法。
- 可选搜索配置：最大深度、是否启用迭代加深、是否启用置换表。

### 6.2 Agent 输出

AI agent 必须输出：

- 一个合法 `Move`。
- 评估分。
- 搜索深度。
- 实际耗时。
- 主要变例或至少一个最优候选走法说明。

如果没有合法走法，应输出困毙状态或认输动作，由裁判逻辑判定结果。

### 6.3 搜索策略

基础版本要求：

- 使用 Alpha-Beta 剪枝或等价的极大极小搜索。
- 在时间限制内完成搜索并返回当前最佳走法。
- 搜索深度可配置，默认至少尝试 2 到 4 ply。
- 当超时临近时返回上一轮完整搜索得到的最佳走法。

增强版本建议：

- 迭代加深：逐层加深，随时保留可用结果。
- 走法排序：优先搜索吃将、吃高价值子、将军、翻高价值暗子、历史最佳走法。
- 置换表：缓存局面哈希、深度、上下界和最佳走法。
- Quiescence Search：在吃子或将军局面继续延伸，降低水平线效应。
- Expectiminimax：对翻开暗子的随机结果显式展开概率分支。

### 6.4 暗子数学期望

评估函数必须纳入暗子的数学期望，不能把暗子简单当作固定棋子。

最低要求：

- 按阵营维护未揭示棋子池。
- 对每个暗子，根据剩余棋子类型概率计算期望价值。
- 若暗子的当前位置限制了当前移动规则，走法生成仍按当前位置对应的原始类型；局面评估按可能真实类型的价值期望。
- 吃暗子后，内部状态应从对应棋子池移除真实类型；如果对某一方不可见，可在该方信息集内用概率更新。

建议公式：

```text
ExpectedValue(hiddenPiece) =
  sum(probability(type | remainingPool, observations) * pieceValue(type, position, phase))
```

评估维度建议包含：

- 基础子力价值。
- 位置价值。
- 将帅安全。
- 机动性。
- 威胁与被威胁。
- 暗子翻开收益或风险。
- 士/象过河后的攻防价值调整。
- 40 步无吃子、长将长捉等规则压力。

## 7. 接口与运行方式

### 7.1 最小可运行模式

必须提供至少一种无需 GUI 的运行方式：

- 控制台对弈：人输入 move，AI 输出 move。
- 脚本对弈：AI vs AI 自动运行固定局数。
- 测试入口：给定棋局 JSON，返回 AI move JSON。

### 7.2 可选 Socket 模式

如果接入课程服务器或真人对弈程序，必须兼容以下外部字段：

```json
{
  "source": "b3",
  "destination": "b4",
  "type": 1,
  "turnStartTime": 1770000000000
}
```

注意：

- 超时以服务器或裁判时间戳为准，客户端时间戳只用于参考。
- AI 不应信任对手客户端传入的非法局面。
- 收到对手 move 后，必须先由 `RuleEngine` 校验，再更新 `GameState`。

## 8. 非功能需求

### 8.1 性能

- 默认每步最多思考 60 秒。
- 单元测试中的 AI 决策应支持较短超时，例如 100 ms 到 2 s。
- 搜索必须可被时间控制中断。
- 整局自动对弈不应出现死循环。

### 8.2 可维护性

- 规则、评估、搜索、通信适配必须分层，避免全部写在一个类中。
- 每个领域类职责清晰，便于单独测试。
- 规则假设必须写入文档或代码注释，尤其是课程文档未完全定义的部分。

### 8.3 可解释性

- 每步 AI 输出应可记录搜索深度、候选走法、评估分和耗时。
- 评估函数权重应集中配置，便于实验调整。
- 暗子概率更新应能通过日志或测试断言验证。

### 8.4 可复现性

- 初始暗子随机打乱必须支持固定随机种子。
- 自动对弈实验应记录随机种子、配置、版本和结果。
- 棋谱记录应足以复盘整局。

## 9. 测试与验收

### 9.1 单元测试

必须覆盖：

- 坐标解析与转换。
- 各类明子合法走法。
- 暗子按当前位置原始类型移动。
- 暗子移动后翻开。
- 原地翻面消耗一回合。
- 吃暗子后的棋子池更新。
- 士/象可过河和离开原限制区域。
- 马腿、象眼、炮架规则。
- 40 步无吃子判和。
- 超时返回当前最佳走法。
- 无合法走法时的困毙或失败处理。

### 9.2 AI 行为测试

必须覆盖：

- 简单必胜局面中优先吃将或将死。
- 明显亏子走法不会在浅层搜索中被优先选择。
- 暗子概率变化会影响评估分。
- 在固定随机种子和固定搜索配置下，AI 输出可复现。
- AI vs AI 能完整运行至少一局，不崩溃、不超时、不产生非法走法。

### 9.3 集成验收

交付时至少满足：

- 程序可以完成一局揭棋对弈流程。
- AI 每步输出合法 `Move`。
- 棋谱可以记录 source、destination、type 和时间戳。
- 项目中存在 5 个以上领域类。
- 有测试证明核心规则和 AI 决策入口可运行。
- 实验报告能说明 AI 辅助分析、设计、编码、测试的过程。

## 10. 推荐任务拆分

后续 agents 可按以下顺序执行：

1. 规则澄清与假设确认：补齐课程文档未明确的揭棋细节，并形成 `RULES.md`。
2. 领域模型实现：`Piece`、`Board`、`GameState`、`Move`、`HiddenPiecePool`。
3. 规则引擎实现：坐标、走法生成、走法校验、胜负和和棋判定。
4. 棋谱与局面序列化：支持 JSON、日志和固定随机种子。
5. 评估函数实现：基础子力、位置、机动性、将帅安全、暗子期望值。
6. 搜索引擎实现：Alpha-Beta、时间控制、迭代加深。
7. Agent 封装：统一 `chooseMove(GameState, SearchConfig)` 决策入口。
8. 运行适配器：控制台、脚本对弈或 Socket 适配。
9. 测试补齐：规则测试、AI 行为测试、AI vs AI 集成测试。
10. 报告材料：架构图、类图、算法说明、测试结果、对战记录。

## 12. 未决规则与建议处理

课程文档明确说明题目定义不完整，先提出并完善问题的小组可获得额外加分。建议开发前整理以下问题并向老师确认：

- 原地翻面是否只能翻己方暗子，还是可翻任意暗子。
- 吃暗子后双方信息可见性如何在联网对战中同步。
- 长将、长捉的精确定义和重复局面计数方式。
- “不应将”错误是否完全允许，还是只由对方下一步吃将处理。
- 暗子随机分配是由服务器统一决定，还是本地 AI 可自行初始化。
- AI 对弈是否必须兼容统一 Socket 协议。

在老师未确认前，程序应把这些假设写入 `RULES.md`，并让规则引擎集中引用，避免散落在搜索和评估代码中。

## 13. 完成定义

本 PRD 对应项目完成条件：

- 能初始化合法揭棋棋局。
- 能接收并校验对手走法。
- 能维护明暗棋子状态和剩余棋子池。
- 能在时间限制内输出合法 AI 走法。
- 评估函数包含暗子数学期望。
- 搜索过程包含剪枝或等价效率优化。
- 能记录可复盘棋谱。
- 能运行自动化测试和至少一局完整对弈演示。
- 能为实验报告提供架构、算法、测试、AI 辅助过程和对战结果材料。
